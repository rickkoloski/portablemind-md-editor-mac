# D24.2 Prompt — Slack-proportional column distribution

You are working on `~/src/apps/md-editor-mac` on branch `feature/d24.2-slack-proportional-columns`. Your job is to replace D24's "lock-in + flex" column distribution with **Q8 narrow-column threshold lock-in + CSS Tables §3.9 slack-proportional** for the flex pool. Same surface as D24 (`Sources/Editor/Renderer/Tables/`), same wiring, same harness — different distribution math.

This closes a D24 regression: at narrow viewports a Date column with natural=86.5pt drops out of D24's lock-in test (its natural exceeds equal-share by ~3pt), gets a flex share below the 60pt floor, snaps to 60pt, and "2026-04-28" wraps mid-string. Literal CSS spec (just min/max + slack) doesn't actually fix it for our use case — TextKit treats `-`/`/`/`.` as soft-break opportunities, so Date min ≈ 36pt and the column STILL ends up below natural under straight slack distribution at narrow viewports. Q8 is the deliberate divergence: pre-lock columns with `max ≤ 120pt` (data fields like dates, IDs, statuses) at max regardless of slack, then slack-distribute the remaining flex pool. Browser/VS Code parity for the long-text case; spreadsheet-intuition for the data-field case.

- A long-text Decision column absorbs nearly all surplus.

The spec's algorithm implements this directly. Don't reinvent or improve without surfacing a `**Question:**`.

## Phase-by-phase guidance

### Phase 1 — Token-split spike + measurement extension

The spike's purpose: validate that "split on ASCII whitespace + soft-break punctuation" produces approximately the same atomic units that TextKit's `byWordWrapping` actually refuses to break. Don't over-engineer — 6-8 fixtures across the dogfooded shapes (prose, dates, URLs, hyphenated paths, inline-code, mixed). If the heuristic is within ~5%, GREEN — proceed. If divergence is large (e.g., TextKit refuses to break on a punctuation character we treat as soft-break), surface a `**Question:**` to CD with the specific divergence + a proposed Q1 refinement.

The cache extension is mechanical: change the value type from `CGFloat` to a `ColumnMeasurement` (struct or named tuple). Existing tests and harness keep compiling because the OLD `distribute` still runs on `max_width` only.

**Behavior unchanged in this phase.** D17 + D24 manual test plans should still produce identical numbers.

### Phase 2 — Algorithm rewrite

Pure-function rewrite. The new algorithm has four regimes (fits / Q8-lock + slack / slack-only / overflow) — each test fixture should explicitly cover its regime. The "Algorithm trace" comment in each test (showing `sum_min`, `sum_max`, Q8 lock decisions, `surplus`, per-col slack share) makes verification easy.

Key correctness checks:
- Q8 pre-lock iterates to fixpoint (1–2 rounds typical). A locked column gets exactly `max`.
- Q8's threshold is a single named constant (default 120pt). Don't hard-code 120 inline; reference the constant.
- A flex-pool column with `slack = 0` gets exactly `min`. Slack-distribution degenerate guard (total_slack=0) splits surplus equally per |flex_cols|.
- `Σ widths = viewport` in the slack regime (within float epsilon).
- Floor clamp doesn't distort slack: it lives outside the math.
- Overflow regime: `Σ widths = Σ min`, may exceed viewport. No NaN/negative.

Don't skip the new regression tests:
- `testNarrowDataColumnLocksAtMaxViaQ8` — the actual Rick reported case.
- `testWideContentColumnFlexesViaSlack` — confirms Q8 doesn't over-trigger.
- `testQ8ThresholdTunable` — same inputs, different threshold values produce different outputs.
- `testQ8IteratesUntilFixpoint` — multi-round Q8 lock behavior.

### Phase 3 — Integration

Wire the new measurement + new distribute call through `computeColumnWidths`. Q8 viewport-cap on `max_width` stays — applied just before passing to `distribute`.

Verify with the harness: drive `set_window_width` to ~250pt, dump `dump_table_layout`, confirm Date.applied == Date.min == ~86.5pt and slack=0.

Snapshot at narrow viewport to capture the visible fix.
