## Why now

D24 shipped a "lock-in + flex" algorithm with a single per-column `natural_width` measurement and a 60pt `min_width` floor. Bug surfaced 2026-05-06 during dogfood (Rick): at narrow viewports a "Date" column carrying "2026-04-28" (≈86.5pt natural) drops out of lock-in (its natural exceeds equal-share by ~3pt at viewport 250pt), gets a flex share below floor, snaps to 60pt floor, and **wraps mid-string** — even though Decision-log columns to its right have ample room.

The base structural fix is well-known: **CSS Tables Module Level 3 §3.9 ("Available Width Distribution")** distributes table width using **two** per-column measurements (`min-content`, `max-content`) and gives every column its `min-content` plus a slack-proportional share of the surplus, where slack = `max − min`. Every browser does this; VS Code preview (Electron/Blink) does this. Adopting it gives polite hyphen-wrap (instead of D24's char-wrap) when columns are forced narrow.

**But literal CSS spec doesn't actually fix Rick's reported case.** TextKit's `byWordWrapping` treats `-`, `/`, `.` as break-after opportunities, so "2026-04-28" has min-content = ~36pt (longest hyphen-segment) — not 86.5pt. Under literal CSS, Decision's enormous max (≈1090pt) gives it ≈95% of slack, leaving Date with only ≈5% of any surplus. At viewport=250 the algorithm produces Date≈40pt (still wraps); at viewport=400 Date≈47pt (still wraps below natural). Browsers do this too — it's the textbook spec output.

The use-case mismatch is real: markdown tables in our editor often hold *structured data* (dates, IDs, statuses, owners) alongside long-form prose. Spreadsheet/data-editor intuition wants narrow data columns to *always* sit at natural width while long-form columns absorb the squeeze. Per **Q8** (added 2026-05-06): pre-lock any column whose `max_content` is below an absolute threshold (default 120pt); slack-distribute over the rest. Tunable threshold so a future setting can dial spec-parity vs data-editor bias.

Per `feedback_exemplar_benchmarking.md`: the divergence from VS Code preview is the decision point. Closing it conceptually (slack distribution for the flex pool) while accepting a deliberate divergence (Q8 threshold lock-in) for the data-editor case. Documented as Q8.

Research artifact: see this conversation's research summary; canonical refs are W3C [css-tables-3 §3.9](https://www.w3.org/TR/css-tables-3/) and the CSSWG [css3-tables-algorithms](https://drafts.csswg.org/css3-tables-algorithms/Overview.src.htm) draft.

## Algorithm

CSS `table-layout: auto` semantics, expressed in terms TK1 can implement.

### Definitions (revised)

- **`min_width(col)`** — `min-content`: the column's widest **unbreakable atom**. Per Q1 below, this is the longest contiguous span TextKit's `byWordWrapping` would refuse to break.
- **`max_width(col)`** — `max-content`: the column's longest single-line shaped width across header + body cells. **(This was D24's `natural_width` — same measurement, renamed for spec parity.)**
- **`slack(col)`** — `max_width(col) − min_width(col)`. Always ≥ 0.
- **`viewport_width`** — text-container width inside the editor, less per-cell framing × column count (same as D24).
- **`min_floor_pt`** — engineering-readability floor (Q3 below). 60pt by default.

### Distribution

```
sum_min = Σ min_width(c)
sum_max = Σ max_width(c)

if sum_max ≤ viewport_width:
    # Everything fits at natural. Lock every column at max.
    for c: width(c) = max_width(c)

else if sum_min ≥ viewport_width:
    # Even the unbreakable atoms don't all fit (Q2 — overflow regime).
    # Per Q2: each column gets its min_width; the table renders wider
    # than the viewport. Cell content within each column wraps via Q9
    # (byWordWrapping → char-wrap last resort) so atoms that exceed
    # min_width(col) still appear losslessly.
    for c: width(c) = min_width(c)

else:
    # Q8 pre-lock: columns whose max_width ≤ narrow_threshold_pt
    # (default 120pt) get max_width and bow out of the slack pool.
    # Iterates: locking shrinks the remaining pool, which can make
    # additional narrow columns "fit" at max. Stops when no more
    # candidates lock (typical: 1–2 iterations for dogfood shapes).
    locked = {}
    pool = viewport_width
    loop:
        new_locks = []
        flex_count = |cols| - |locked|
        flex_pool = pool - Σ width(c) for c in locked
        for c in cols not in locked:
            if max_width(c) ≤ narrow_threshold_pt
               and max_width(c) ≤ flex_pool:
                new_locks.append(c)
        if new_locks empty: break
        for c in new_locks:
            width(c) = max_width(c)
            locked.add(c)

    # Slack-proportional on the flex pool (post-Q8). Columns with
    # slack=0 (unbreakable atoms == longest line) get exactly min_width
    # and bow out of the slack share automatically.
    flex_cols = cols - locked
    if flex_cols is empty:
        # Q8 locked everything; done.
    else:
        flex_pool = viewport_width - Σ width(c) for c in locked
        flex_sum_min = Σ min_width(c) for c in flex_cols
        surplus = flex_pool - flex_sum_min
        total_slack = Σ slack(c) for c in flex_cols
        for c in flex_cols:
            share = total_slack > 0
                ? slack(c) / total_slack * surplus
                : surplus / |flex_cols|     # all-equal-min-max degenerate
            width(c) = min_width(c) + share
```

The floor (`min_floor_pt`) is applied as a **post-pass clamp** per Q3:

```
for c: width(c) = max(width(c), min_floor_pt)
# If the floor pushed Σ widths past viewport, accept overflow per Q2.
```

D24's "natural ≤ equal_share" lock-in test is replaced by Q8's "max ≤ absolute threshold" lock-in. The remaining flex pool gets canonical CSS slack-proportional distribution. The 60pt floor's role narrows to "absolute minimum readable width" — for any non-pathological cell the floor never fires.

## Decision log

| Date | Decision | Decided by |
|---|---|---|
| 2026-05-06 | **Q1 — Definition of `min-content` for markdown cells.** A cell's min-content is the longest span TextKit's `byWordWrapping` mode refuses to break, measured at `Typography.boldFont` (header) or `Typography.baseFont` (body). Implementation: walk the cell's plain-text representation (after the same single-line + escape-undo extraction the renderer uses), find the widest contiguous run that has no soft-break opportunity (no whitespace, no `-`, no `/`, no `.`, no other ASCII punctuation TextKit treats as a wrap point), measure that run's CT-shaped width. Cell-level min-content is that width; column-level min-content is the max across the column's cells. Inline-code spans (markdown `\`code\``) treat as **atomic** (no internal break opportunities) — matches how cells render today. **Why this approximation:** TextKit's actual break-opportunity table is implementation-defined and CJK / emoji / RTL adds wrinkles we don't dogfood. ASCII-punctuation-as-soft-break covers every dogfooded markdown shape (URLs, paths, identifiers, prose, dates) and is forward-compatible with later refinement. | RAK |
| 2026-05-06 | **Q2 — Narrow regime (`Σ min_width ≥ viewport`).** Each column gets its min-content; the table renders wider than the viewport. **No horizontal scroll** — the editor's NSScrollView remains vertical-only. Cell content wraps via Q9 (`byWordWrapping` + char-wrap last resort) inside its column, so atoms exceeding `min_width` still appear losslessly even though the column is "at its minimum." This matches browser behavior (table overflows its container; user gets horizontal scroll IF the parent permits it; we don't permit it, accept the visual). Phase 4 may add a horizontal scroll affordance later if dogfood surfaces a pain point — out of scope for D24.2. | RAK |
| 2026-05-06 | **Q3 — `min_floor_pt = 60`.** Kept as a final post-pass clamp protecting against pathological tiny inputs (e.g., a column whose min-content is a single `.` glyph at ~5pt). The floor applies AFTER slack distribution, so it doesn't distort the algorithm. If applying the floor causes Σ widths to exceed viewport, accept overflow per Q2 (floor wins over fitting). For typical markdown content (any cell with whitespace or punctuation), `min_width` is already larger than 60pt so the floor never fires; this is purely a safety net. | RAK |
| 2026-05-06 | **Q4 — Header vs body rows.** Same algorithm, no special-casing. Header participates in the column's `min_width` and `max_width` measurements like any other row. (Same as D24 Q5.) | RAK |
| 2026-05-06 | **Q5 — Caching.** Extend `TableNaturalWidthCache` to hold a `(min, max)` tuple per content-hash entry. Same hash key (column-content-hash); same invalidation rules. Cost: one extra CGFloat per cache entry; negligible. Alternative considered: separate `min` cache. Rejected — both measurements are computed in the same pass per column, no benefit to splitting. | RAK |
| 2026-05-06 | **Q6 — Q9 (`byWordWrapping`) interaction.** Q9 stays. Symmetry: the algorithm gives every column at least its min, which is the algorithm's promise that no column will char-wrap inside its primary atoms; if char-wrap appears, it's because the cell's content has an atom genuinely wider than `min_width(col)` (impossible by construction; would only happen on data corruption or measurement skew). | RAK |

Iterates to fixpoint so cascading locks resolve in 1–2 rounds. Slack-proportional distribution applies to the remaining flex columns. **Why:** literal CSS (just min/max + slack) doesn't actually fix the regression Rick reported — TextKit treats `-`/`/`/`.` as soft-break opportunities, so Date min ≈ 36pt and Decision min ≈ 70pt; sum_max for the Decision-log shape gives Decision ≈95% of slack, leaving Date with too little surplus to reach natural width even at moderate viewports. Spreadsheet/data-editor convention; deliberate divergence from strict CSS for our use case. **Threshold tunable:** start at 120pt as a heuristic; expose as a setting later if dogfood surfaces a need to dial spec-parity vs data-editor bias. The threshold being absolute (not a ratio) is intentional — what makes a column "narrow" is the absolute width of its atomic content, not its relation to other columns. | RAK |

## Edge cases

- **Date column at narrow viewport** (the regression case) — Date max ≈ 86.5pt < `narrow_threshold_pt = 120pt` → **Q8 pre-locks Date at max** (not via slack=0; literal CSS would NOT lock Date because Date min < max). Decision column flexes the remaining viewport. No wrap on Date for any non-pathological viewport.
- **Decided by column** ("RAK", max ≈ 28pt) — also Q8-locked at max; same path.
- **All-narrow data table** (every column < 120pt) — Q8 locks every column at max → Σ widths = Σ max ≤ viewport. Effectively the same as the "fits naturally" branch; falls through cleanly.
- **Many narrow + one wide** (Decision-log shape) — Q8 locks the narrow ones at max, slack distribution applies to the wide column → wide column gets `min` + entire surplus.
- **Pathological super-long token** (long URL, base64) — column's `max_width` capped at viewport per **Q8 (D24 carry-over — same name, different decision)**: that's the viewport cap from D24's Q8, distinct from D24.2's narrow-threshold Q8. The column is wide (`max_width = viewport`, much greater than threshold) → falls into the slack pool. Its `min_width` is the longest unbreakable run; URL wraps at `-`/`/`/`.` Q9-style within the column's allotted width.
- **All-equal-min-max columns at viewport pressure** — every column has slack 0 in the flex pool → surplus distributes equally per `|flex_cols|` (degenerate path). Q8 locks any below threshold first, so this path is rare.
- **Single column** — if max ≤ threshold, Q8 locks at max; table sits at natural with whitespace to its right. If max > threshold and exceeds viewport, slack distribution gives the only column the full pool.
- **Empty table** — no columns; algorithm returns empty per existing degenerate-input handling.
- **Header-only (no body)** — only header cells contribute; min/max derived from header text alone.

**Q8 carry-over vs Q8 (D24.2):** D24's spec used "Q8" for the viewport-cap on natural width (preventing super-long tokens from forcing the table past viewport). D24.2's Q8 is a separate decision — the narrow-column threshold lock-in. Both stay; the viewport-cap survives unchanged. Only the algorithm body for distributing the non-cap-pre-clamped widths changes. Where ambiguous, this spec calls them "D24 Q8 (viewport cap)" and "D24.2 Q8 (narrow-threshold lock)" respectively.

## Acceptance criteria

1. **Date column regression closed.** "2026-04-28" renders on a single line at every editor viewport above the overflow regime. Q8 pre-locks the Date column at max ≈ 86.5pt across the entire normal-viewport range.
2. **Visual parity-of-spirit with VS Code preview** on the canonical 3-col Decision-log fixture: narrow cols hug content; long-text col absorbs all surplus. Cross-checked at 3 viewports (narrow / medium / wide). Note: literal pixel parity with VS Code is NOT a goal — Q8 is a deliberate divergence (see decision log).
3. **Pathological URL renders correctly:** a super-long URL with internal break opportunities (`-`, `/`, `.`) renders inside a column wide enough to flex (URL column's max > narrow_threshold_pt → goes to slack pool); URL wraps at break points within the allotted width.
4. **Overflow regime (`Σ min ≥ viewport`) reachable** without crash, NaN, or negative widths. Every column drops to its min-content; the table extends past the viewport (Q2). No horizontal scrollbar.
5. **Q8 threshold tunable.** `narrow_threshold_pt` is a single named constant (not magic) so future settings UI can dial it. Default 120pt; document the decision in code.
6. **Algorithm performance:** measurement adds one CT-shaped pass per cell to compute min-content. 10-row × 4-col table: <50ms first-render, <5ms resize (cache hits).
7. **All 15 D24 unit tests rewritten** with new expected values. New tests added: (a) "narrow data column under viewport pressure stays at max via Q8"; (b) "wide-content column under same viewport flexes via slack." Viewport-sweep invariant test covers regimes (`fits / Q8-lock + slack / slack-only / overflow`).
8. **D24 manual test plan rerun GREEN** with new column-width numbers. Narrow data columns locked at max via Q8; long-text columns flex via slack. Document the algorithm-shift in COMPLETE doc.
9. **Harness `dump_table_layout` extended** to surface per-column `min_width`, `max_width`, `slack`, `q8_locked` (bool), and overall regime. Drivers can assert on regime + per-column lock state.

## Out of scope (deferred)

- **Horizontal scroll for the overflow regime** (Q2). Editor stays vertical-only; the user-visible result of `Σ min ≥ viewport` is "table runs off the right edge." Phase a horizontal scroll affordance later if dogfood requires.
- **CJK / RTL / emoji-aware break-opportunity rules** — Q1's ASCII-punctuation-soft-break heuristic is the v1. We don't dogfood other scripts heavily; revisit if a real user surfaces it.
- **User-resizable columns** — already deferred from D24.
- **Inline-image intrinsic widths** — already deferred from D24.
- **Cross-row colspan / rowspan** — GFM doesn't support them; not in our markdown surface.

## Risks

1. **Token-splitting fidelity drift.** Q1's "ASCII punctuation as soft break" is a *heuristic* approximation of TextKit's actual break-opportunity table. If they diverge — e.g., TextKit refuses to break on a character we treat as a soft break — the column's `min_width` is underestimated and char-wrap fires unexpectedly inside the algorithm-allotted width. Mitigation: phase 1 includes a small spike that compares Q1's atom-width to `byWordWrapping`'s actual minimum-line width on a stress fixture; if they diverge, document and refine. Cheap; ~30 min.
2. **Performance regression.** Adding a per-cell min-content measurement doubles measurement cost. Mitigation: same content-hash cache covers both; first-render budget is the same; resize budget unchanged (cache hits).
3. **Q3 floor edge case.** If a column's `min_width` < 60pt and the floor pushes the column over `min`, slack distribution math could under-credit other columns. Mitigation: post-pass clamp lives outside the slack math; slack distribution always uses raw `min`. The floor only affects very narrow content (single-glyph columns), which dogfood doesn't exercise.
4. **Test fixture rewrites.** All 15 D24 unit tests change in expected values. Risk: I introduce a regression while transcribing. Mitigation: derive expected values from the published CSS algorithm and double-check with a hand-trace per test case in the COMPLETE doc.
5. **D24 manual test plan numbers shift.** Documented widths in `d24_responsive_table_columns_manual_test_plan.md` (e.g., Decision = 705.4pt at viewport=920) will change because the new algorithm distributes differently. Plan: update those numbers in the manual test plan as part of phase 4 close-out.
