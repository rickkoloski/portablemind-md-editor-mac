## 1. Problem Statement

D1 validated TextKit 2 via a disposable spike at `spikes/d01_textkit2/`. The spike intentionally skipped architecture polish, code quality, and the module boundaries we need for feature work. To enable any further feature deliverable (toolbar, folder tree, Submit UX, document-type extensibility, etc.), the project needs a **real Xcode project at `apps/md-editor-mac/` with an intentional module structure**, the **five D1 findings addressed** (so we don't inherit spike debt), and all **engineering standards honored from the first commit** (so we don't retrofit later).

This is Option B from the D2 scope discussion: *functional* promotion to real project, no packaging/deployment work. Sparkle, DMG, notarization are D3.

## 3. Design

### Approach

Create a single Xcode project at `apps/md-editor-mac/` (alongside the existing `docs/` and `spikes/` directories) generated by `xcodegen` from a committed `project.yml`. Source is organized into **modules by purpose**, with each module mapping onto one of the nine cross-OS abstractions from `docs/stack-alternatives.md`. The spike's behavior is lifted in; the spike's bugs are fixed during the lift.

Not every abstraction needs code at D2. Some modules only need a directory + a protocol/stub file so the boundary exists for later deliverables to extend. The discipline: **the module directory exists now; the implementation arrives with its feature deliverable.**

### Key components and module layout

```
apps/md-editor-mac/
├── project.yml                       xcodegen source
├── Info.plist                        full per standard §1.3
├── MdEditor.xcodeproj/               generated, committed
├── Sources/
│   ├── App/                          SwiftUI App, window scene, file picker
│   ├── DocumentTypes/                Registry + MarkdownDocumentType (v1 impl)
│   ├── Editor/                       TextKit 2 hosting + renderer
│   │   └── Renderer/                 Markdown → attributes, cursor-on-line
│   ├── Files/                        NSFilePresenter, external-edit reconcile
│   ├── Accessibility/                AccessibilityIdentifiers constants
│   ├── Support/                      Typography, shared utility types
│   ├── Handoff/                      (stub) Submit / handoff protocol
│   ├── Toolbar/                      (stub) ToolbarAction model
│   ├── Keyboard/                     (stub) KeyboardBindings
│   ├── Settings/                     (stub) AppSettings schema
│   └── Localization/                 Localizable.xcstrings (English only at v1)
└── UITests/
    └── LaunchSmokeTests.swift        accessibilityIdentifier-based query
```

**Which modules have real code at D2:**
- `App/`, `DocumentTypes/`, `Editor/`, `Files/`, `Accessibility/`, `Support/`, `Localization/` — all get real code.
- `Handoff/`, `Toolbar/`, `Keyboard/`, `Settings/` — only a `README.md` + a protocol or model-struct stub that names the abstraction. No implementation until their respective feature deliverable.

### Renaming from spike

Spike type → new project type mapping:

| Spike | New |
|---|---|
| `SpikeApp.swift` | `App/MdEditorApp.swift` |
| `EditorContainer.swift` | `Editor/EditorContainer.swift` |
| `LiveRenderTextView.swift` (implicit in spike) | `Editor/LiveRenderTextView.swift` (explicit) |
| `MarkdownRenderer.swift` | `Editor/Renderer/MarkdownRenderer.swift` (+ fixes for findings #3, #4) |
| `CursorLineTracker.swift` | `Editor/Renderer/CursorLineTracker.swift` (+ fix for finding #2) |
| `ExternalEditWatcher.swift` | `Files/ExternalEditWatcher.swift` |
| `SpikeTypes.swift` | split into `Support/Typography.swift` + `Support/RenderTypes.swift` |

No file is copied verbatim; each is revised during the move to honor standards (no `.layoutManager`, identifiers wired, types namespaced to the app not to "Spike").

### Document-type registry (first real abstraction)

```swift
protocol DocumentType {
    static var fileExtensions: [String] { get }
    func render(_ source: String) -> RenderResult
}

struct MarkdownDocumentType: DocumentType {
    static let fileExtensions = ["md", "markdown"]
    private let renderer = MarkdownRenderer()
    func render(_ source: String) -> RenderResult { renderer.render(source) }
}

final class DocumentTypeRegistry {
    static let shared = DocumentTypeRegistry()
    private init() { register(MarkdownDocumentType.self) }
    func type(for fileURL: URL) -> DocumentType? { /* match by extension */ }
    func register(_ type: DocumentType.Type) { /* ... */ }
}
```

This is the minimum abstraction that satisfies Principle 3 ("markdown today, JSON/YAML tomorrow") without overbuilding. It also forces D2 to expose `RenderResult` as a shared type, surfacing any hidden markdown-specific assumptions in the spike code.

### Fixes for the five D1 findings

- **#1 (`.layoutManager`)** — the retained startup assertion in `EditorContainer` is strengthened to only check `textLayoutManager`, with a comment header marking the trap. No other file references `.layoutManager`.
- **#2 (pre-collapse on initial render)** — `CursorLineTracker` gets a `collapseAllDelimiters(in: NSTextView)` method; `EditorContainer.renderCurrentText` calls it after applying assignments, before the first `updateVisibility`.
- **#3 (inline-code range)** — `MarkdownRenderer`'s `SourceLocation → NSRange` conversion is fixed with per-parse line-start caching; `InlineCode` is tested explicitly in a unit test with known expected NSRanges.
- **#4 (code-block range + fence delimiters)** — `visitCodeBlock` tags opening and closing fences as delimiters; range conversion is corrected via the same line-start fix from #3. Unit tests cover multi-line ranges.
- **#5 (`accessibilityIdentifier`)** — every `NSView` created in the app is given an identifier from `Accessibility/AccessibilityIdentifiers.swift` at creation time.

## 4.5 Validation approach

Per Native lifecycle §6 (Phases 2–4 skip justification) — D2 is still pre-feature work, tested via **Phase 1 exploratory** demonstration plus a **single XCUITest** for smoke + accessibility-identifier-based query validation. Full Playwright-equivalent test infrastructure is premature before any real feature ships.

**Evidence artifacts (same shape as D1):**
- A trimmed demo run — the D1 demo steps whose outcome *changes* with the findings fixes: Step C (initial render clean), Step F/J (inline code + fenced code block precision), plus a launch + identifier-query confirmation. Screen recording + transcript in `apps/md-editor-mac/evidence/d02/`.
- Xcode console log captured for the demo session.
- System info snapshot.
- `xcodebuild test` output proving the one XCUITest passes by `accessibilityIdentifier`-based query.

No Instruments measurement — the D1 subjective smoothness and the fact that D2 changes nothing computationally-significant mean repeating Instruments would add no information.

Spec per `engineering-standards_ref.md` §2.1: the XCUITest MUST use identifier-based queries, NOT element-type queries. This both tests the app and verifies standard §2.1 is being honored.

## 5. Out of Scope

Each item below is deliberately excluded from D2. Specs that follow will own them.

- **Sparkle auto-update wiring** — D3 (deployment readiness).
- **DMG packaging and create-dmg / hdiutil tooling** — D3.
- **Developer ID signing and notarization** — D3 (also gated on Apple Developer Program enrollment per `engineering-standards_ref.md` §1.4).
- **EdDSA key generation for Sparkle appcast** — D3.
- **Any user-facing feature beyond the spike's "open a file, see live-render"** — toolbar, folder tree / sidebar, Submit/Handoff UI, Preferences window, multi-document, tabs, dark mode polish, anything else each earns its own deliverable.
- **Real implementations of `Handoff/`, `Toolbar/`, `Keyboard/`, `Settings/`** — directories and stub files only at D2.
- **Performance optimization beyond what the spike already demonstrated** — the D1 subjective smoothness on 284 lines is the D2 baseline; D2 is not expected to improve or regress it.
- **Localization beyond English** — `Localizable.xcstrings` exists with English-only strings; other locales arrive when they do.
- **Typography evolution beyond the spike** — body font remains as it was in D1 for D2; the "switch to proportional body" UX change is a separate, intentional deliverable because it alters the product's feel.

## 6. Open Questions

1. **Minimum macOS version** — spike built against 14.0 (Sonoma) cleanly. Do we lock `LSMinimumSystemVersion = 14.0` in D2, or raise to 15.0 for any TextKit 2 affordances Sonoma lacks? Recommendation: **stay at 14.0** unless a D2 finding forces higher. Resolve early in implementation; record in Info.plist.
2. **Single-module vs multi-module SPM** — single module (one target, module directories as Swift-file groupings) is simpler for D2; multi-module (each directory is its own SPM library target) enforces abstraction boundaries at the compiler level. Recommendation: **single module** for D2; re-evaluate if we see cross-module coupling creep in D3–D5.
3. **Commit the generated `.xcodeproj`** or require xcodegen to regenerate? D1 committed it. Recommendation: **continue committing** for reproducibility; re-evaluate if git noise from the generated file becomes friction.
4. **Retire or retain `spikes/d01_textkit2/`?** The spike is useful as a reference (a known-good minimal version). Recommendation: **retain with a `README.md` banner** marking it frozen; don't update it when D2 fixes land. Delete only if it becomes confusing.
5. **Typography baseline** — spike used monospace base font; our priority-1 audience (Word/Docs users) expects proportional. The Out-of-Scope list above keeps D2 identical to D1 for typography; is that right, or do we want the proportional switch in D2? Recommendation: **defer the switch to a dedicated deliverable** (the feel change is too load-bearing to slip into scaffolding work).
