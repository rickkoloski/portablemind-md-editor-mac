## Why now

Two motivations, paired by API surface:

1. **Self-cleaning smoke tests.** D23 phases 2–5 each created an artifact in Rick's PM tenant (`/Sales & Marketing/d23-smoke-...md`, `d23-newfile-...md`, etc.) requiring manual cleanup in the web UI after every smoke run. Without a programmatic delete + the ability to create/destroy a scratch directory, the smoke tests pollute real tenant data on every iteration. Rick called this out 2026-05-08 — accretive build of these capabilities is the right path.

2. **Standalone user value.** Delete file is a normal-day need; users currently have no in-editor way to remove unwanted PM files. Directory create + delete are similarly normal-day ops. Both are listed in D23's deferred-follow-ups and would have shipped eventually; pulling them forward costs almost nothing because D23 already established the connector + API-client + sheet patterns they reuse.

What this deliverable explicitly does NOT do (deferred to D26 and beyond):
- Directory rename (PATCH llm_directory.name)
- Directory move (PATCH llm_directory.parent_path; cascade semantics)
- Trash semantics / undo on delete (v1 is hard-delete with confirmation)
- Multi-select delete
- Drag-drop ops

## Scope

In scope:

| Operation | Trigger | Connector method | Notes |
|---|---|---|---|
| **Delete file** | Sidebar context menu → Delete… on PM file row | `Connector.deleteFile(_:)` | NSAlert confirmation ("Delete 'foo.md'? This can't be undone."). Cascade-closes any open tab. |
| **Create directory** | Sidebar context menu → New Folder… on a directory row (or the root) | `Connector.createDirectory(in:name:)` | Small input modal (variant of RenameSheet — single field). |
| **Delete directory** | Sidebar context menu → Delete… on PM directory row | `Connector.deleteDirectory(_:)` | NSAlert confirmation with **child count** in the message ("Delete 'foo'? This will also delete N file(s)."). Cascade-closes any open tabs whose nodes lived inside. |

Server endpoints (verified 2026-05-08 in `LlmDirectoriesController` and `LlmFilesController`):
- `POST /api/v1/llm_directories` — create.
- `DELETE /api/v1/llm_directories/:id` — destroy (cascade).
- `DELETE /api/v1/llm_files/:id` — destroy.

Out of scope (deferred):
- **Directory rename / move** — D26.
- **Trash / undo** — separate higher-stakes design conversation; v1 is hard-delete with confirmation.
- **Multi-select** — same as D23 deferred list.
- **Local connector implementations** — provided for parity but not surfaced in the UI yet (Local sidebar doesn't surface create-folder/delete in v1; can ship later).
- **Cross-tenant variants** — same Q6 scope as D23.

## UX flows

### Delete file

1. Right-click a PM file row → "Delete…".
2. NSAlert: "Delete '`<filename>`'?" with informativeText "This can't be undone." Buttons: **Cancel** (default), **Delete**.
3. On Delete: `connector.deleteFile(node)` → server returns 200 → cascade-close any open tab whose `connectorNode.id == node.id`.

### Create directory

1. Right-click a PM directory row (or the connector's root) → "New Folder…".
2. CreateDirectorySheet opens: title + filename text field + Save / Cancel + inline error.
3. On Save: `connector.createDirectory(in: parent, name: filename)` → server returns 200 with the new directory record → splice into the connector's cached tree (Q3) → close sheet.

### Delete directory

1. Right-click a PM directory row → "Delete…".
2. NSAlert: "Delete '`<dirname>`'?" with informativeText "This will also delete `<N>` file(s) inside. This can't be undone." Buttons: **Cancel** (default), **Delete**.
3. On Delete: `connector.deleteDirectory(node)` → server cascade-deletes → close any open tab whose connectorNode is inside the deleted dir → splice removal into the connector's cached tree.

## Decision log

| Date | Decision | Decided by |
|---|---|---|
| 2026-05-08 | **Q1 — Confirmation pattern.** Use stock `NSAlert` for delete confirmations (file + directory). Same modal pattern D14's "Save unsupported" alert used; reliable, accessible, no custom UI work required. The CreateDirectorySheet is a SwiftUI sheet (consistent with the SaveAsSheet / RenameSheet / MoveSheet pattern from D23). | RAK |
| 2026-05-08 | **Q2 — Hard delete vs trash.** v1 is hard-delete with confirmation. The server's `DELETE /api/v1/llm_files/:id` already removes the record + ActiveStorage blob; no Trash table or recovery semantics exist server-side. Adding undo / trash would be a bigger design conversation crossing the API boundary; confirmation dialog is the safety net for v1. | RAK |
| 2026-05-08 | **Q3 — Cache mutation.** After a successful create/delete, splice the change into the connector's cached tree directly (in-memory `asyncChildren` mutation on `ConnectorTreeViewModel`) rather than refetching the parent's children. Same `TODO-D23-tree-splice` follow-up that's been carried since D23 phase 2 — D23.1 is the right time to land it because the UI surface needs the splice for create-folder feedback to be acceptable. v1 splice scope: the renamed/moved/created/deleted node only (D23-phase-2/3/4/5 surfaces benefit too as a bonus). | RAK |
| 2026-05-08 | **Q4 — Cascade-close open tabs on delete.** When deleting a file (or a directory containing open files), iterate `tabs.documents` and close any tab whose `connectorNode.id` matches the deleted file (or whose `connectorNode.path` is under the deleted directory). The buffer is discarded; the user already confirmed delete. No "save first?" prompt. | RAK |

## Acceptance criteria

1. **Delete file** via context menu confirms, deletes on server, cascade-closes any open tab.
2. **Create directory** via context menu opens CreateDirectorySheet, creates on server, sidebar tree shows the new folder without manual reload (Q3 splice).
3. **Delete directory** via context menu confirms (with child count surfaced), deletes on server with cascade, removes from sidebar tree, cascade-closes any open tabs inside.
4. **Sidebar tree splice** works for create + delete (and as a bonus, fixes the lingering `TODO-D23-tree-splice` for create/rename/move from D23).
5. **Self-cleaning smoke pattern** demonstrated: harness recipe creates a `/d23-test-scratch-<TS>/` directory, runs the D23 file CRUD smoke set inside it, then deletes the directory; PM tenant has no residue.
6. **Harness actions** added: `pm_delete_file`, `pm_create_directory`, `pm_delete_directory`.
7. **Inline / confirmation errors** surface correctly: 401/403 → "Write denied"; 404 → "Already deleted"; etc.

## Risks / open implementation questions

1. **Cascade-close open tabs.** Reliably finding tabs inside a deleted directory needs path-prefix matching (`tab.connectorNode.path.hasPrefix(deletedDir.path + "/")`). Edge: a directory's path doesn't have a trailing slash; need to be careful not to match `/projects-old` when deleting `/projects`. Test: explicit trailing-`/` boundary check.
2. **Tree splice correctness.** `ConnectorTreeViewModel.asyncChildren` is a published dict keyed by parent path. Splice = mutate the array at the parent key + nudge `objectWillChange`. Need to handle both PortableMind (async) and Local (sync) — Local may never have populated `asyncChildren` so its tree refresh path is different. v1 scope for splice: PortableMind only; Local refreshes via existing `FolderTreeWatcher`.
3. **Server deletes with permissions.** Like D19's writeForbidden path, delete may 401/403; surface as a normal error in the confirmation alert flow.
4. **Empty directory enumeration before delete.** The "child count" in the delete-directory confirmation requires knowing how many files are inside. Use the connector tree's already-loaded counts or call `GET /api/v1/llm_files?directory_path=...&limit=-1` for an authoritative count. Slightly more network but keeps the confirmation honest.
