## Why now

D24 shipped a "lock-in + flex" algorithm with a single per-column `natural_width` measurement and a 60pt `min_width` floor. Bug surfaced 2026-05-06 during dogfood (Rick): at narrow viewports a "Date" column carrying "2026-04-28" (≈86.5pt natural) drops out of lock-in (its natural exceeds equal-share by ~3pt at viewport 250pt), gets a flex share below floor, snaps to 60pt floor, and **wraps mid-string** — even though Decision-log columns to its right have ample room.

The structural fix is well-known: **CSS Tables Module Level 3 §3.9 ("Available Width Distribution")** distributes table width using **two** per-column measurements (`min-content`, `max-content`) and gives every column its `min-content` plus a slack-proportional share of the surplus, where slack = `max − min`. A column whose min equals its max (an unbreakable atom like "2026-04-28") has zero slack, gets zero surplus, and stays at natural — exactly Rick's expectation. Every browser does this; VS Code preview (Electron/Blink) does this. Our D24 algorithm conflates `max` with `natural` and has no `min` concept, which is why it fails the case.

Per `feedback_exemplar_benchmarking.md`: the divergence from VS Code preview is the decision point. Closing it.

Research artifact: see this conversation's research summary; canonical refs are W3C [css-tables-3 §3.9](https://www.w3.org/TR/css-tables-3/) and the CSSWG [css3-tables-algorithms](https://drafts.csswg.org/css3-tables-algorithms/Overview.src.htm) draft.

## Algorithm

CSS `table-layout: auto` semantics, expressed in terms TK1 can implement.

### Definitions (revised)

- **`min_width(col)`** — `min-content`: the column's widest **unbreakable atom**. Per Q1 below, this is the longest contiguous span TextKit's `byWordWrapping` would refuse to break.
- **`max_width(col)`** — `max-content`: the column's longest single-line shaped width across header + body cells. **(This was D24's `natural_width` — same measurement, renamed for spec parity.)**
- **`slack(col)`** — `max_width(col) − min_width(col)`. Always ≥ 0.
- **`viewport_width`** — text-container width inside the editor, less per-cell framing × column count (same as D24).
- **`min_floor_pt`** — engineering-readability floor (Q3 below). 60pt by default.

### Distribution

```
sum_min = Σ min_width(c)
sum_max = Σ max_width(c)

if sum_max ≤ viewport_width:
    # Everything fits at natural. Lock every column at max.
    for c: width(c) = max_width(c)

else if sum_min ≥ viewport_width:
    # Even the unbreakable atoms don't all fit (Q2 — narrow regime).
    # Per Q2: each column gets its min_width; the table renders wider
    # than the viewport. Cell content within each column wraps via Q9
    # (byWordWrapping → char-wrap last resort) so atoms that exceed
    # min_width(col) still appear losslessly.
    for c: width(c) = min_width(c)

else:
    # Slack-proportional: every col gets min, surplus distributes to
    # slack. Columns with slack=0 (unbreakable atoms == longest line)
    # get exactly min_width and bow out automatically.
    surplus = viewport_width - sum_min
    total_slack = Σ slack(c)
    for c:
        share = total_slack > 0
            ? slack(c) / total_slack * surplus
            : surplus / |cols|         # all-equal-min-max degenerate
        width(c) = min_width(c) + share
```

The floor (`min_floor_pt`) is applied as a **post-pass clamp** per Q3:

```
for c: width(c) = max(width(c), min_floor_pt)
# If the floor pushed Σ widths past viewport, accept overflow per Q2.
```

The lock-in iteration from D24 is **gone**; columns whose slack is 0 stay at natural automatically. The 60pt floor's role narrows to "absolute minimum readable width" (cells whose min-content is already very small don't get clamped further; the floor only protects against pathological tiny inputs).

## Decision log

| Date | Decision | Decided by |
|---|---|---|
| 2026-05-06 | **Q1 — Definition of `min-content` for markdown cells.** A cell's min-content is the longest span TextKit's `byWordWrapping` mode refuses to break, measured at `Typography.boldFont` (header) or `Typography.baseFont` (body). Implementation: walk the cell's plain-text representation (after the same single-line + escape-undo extraction the renderer uses), find the widest contiguous run that has no soft-break opportunity (no whitespace, no `-`, no `/`, no `.`, no other ASCII punctuation TextKit treats as a wrap point), measure that run's CT-shaped width. Cell-level min-content is that width; column-level min-content is the max across the column's cells. Inline-code spans (markdown `\`code\``) treat as **atomic** (no internal break opportunities) — matches how cells render today. **Why this approximation:** TextKit's actual break-opportunity table is implementation-defined and CJK / emoji / RTL adds wrinkles we don't dogfood. ASCII-punctuation-as-soft-break covers every dogfooded markdown shape (URLs, paths, identifiers, prose, dates) and is forward-compatible with later refinement. | RAK |
| 2026-05-06 | **Q2 — Narrow regime (`Σ min_width ≥ viewport`).** Each column gets its min-content; the table renders wider than the viewport. **No horizontal scroll** — the editor's NSScrollView remains vertical-only. Cell content wraps via Q9 (`byWordWrapping` + char-wrap last resort) inside its column, so atoms exceeding `min_width` still appear losslessly even though the column is "at its minimum." This matches browser behavior (table overflows its container; user gets horizontal scroll IF the parent permits it; we don't permit it, accept the visual). ## Edge cases

- **Date column at narrow viewport** (the regression case) — Date min == max == 86.5pt → slack = 0 → gets exactly 86.5pt regardless of viewport. Decision column absorbs all surplus. No wrap.
- **Pathological super-long token** (long URL, base64) — column's `max_width` capped at viewport per Q8 (D24 carry-over). Column's `min_width` is the longest unbreakable run within the cell content. If the URL has hyphens/slashes (typical), `min_width` is the longest segment between break opportunities — much smaller than the full URL. Column gets `min_width` + slack share; URL wraps at `-`/`/`/`.` Q9-style.
- **All-equal-min-max columns** (every column is short atomic content) — every column has slack 0 → surplus distributes equally per `|cols|` (degenerate path in the algorithm). Effectively centers the table content; surplus eats per-column margin. Acceptable.
- **Single column** — surplus distributes to its slack; if slack == 0, column at min, table sits at natural width with whitespace to its right. (Same as D24 single-column behavior.)
- **Empty table** — no columns; algorithm returns empty per existing degenerate-input handling.
- **Header-only (no body)** — only header cells contribute; min/max derived from header text alone.

## Acceptance criteria

1. **Date column regression closed.** At every editor viewport tested, "2026-04-28" renders on a single line in any column whose only content is short atomic strings.
2. **Visual parity with VS Code preview** on the canonical 3-column Decision-log fixture: short cols hug atoms; long-text col absorbs all surplus. Cross-checked at 3 viewports (narrow / medium / wide).
3. **Pathological URL renders correctly:** a single super-long URL with internal break opportunities (`-`, `/`, `.`) wraps within its column at those break points; the column gets its `min_width` (longest hyphen-separated segment) plus a slack share.
4. **`Σ min ≥ viewport` regime is reachable** without crash, NaN, or negative widths. At a sufficiently narrow viewport, every column drops to its min-content; the table extends past the viewport (overflow visible — Q2). Regression: confirm no horizontal scrollbar appears.
5. **Algorithm performance:** measurement adds one CT-shaped pass per cell to compute min-content. For a 10-row × 4-col table, total measurement time stays under 50ms first-render and <5ms on resize (cache hits). Same budget as D24.
6. **All 15 D24 unit tests rewritten** with new expected values per slack-proportional algorithm. New test added: "unbreakable atom under viewport pressure" (the regression case). Viewport-sweep invariant test covers the three regimes (`fits / surplus / overflow`).
7. **D24 manual test plan rerun GREEN** with the algorithm change; columns that were "locked at natural" before are still rendered correctly. Document any column-width changes that result from algorithm change vs D24 (specifically: short atomic columns that flexed under D24 now sit at natural).
8. **Harness `dump_table_layout` extended** to surface per-column `min_width`, `max_width`, `slack`, and the regime (`fits` / `surplus` / `overflow`). Test drivers can assert on regime + slack distribution.

## Out of scope (deferred)

- **User-resizable columns** — already deferred from D24.
- **Inline-image intrinsic widths** — already deferred from D24.
- **Cross-row colspan / rowspan** — GFM doesn't support them; not in our markdown surface.

## Risks

1. **Token-splitting fidelity drift.** Q1's "ASCII punctuation as soft break" is a *heuristic* approximation of TextKit's actual break-opportunity table. If they diverge — e.g., TextKit refuses to break on a character we treat as a soft break — the column's `min_width` is underestimated and char-wrap fires unexpectedly inside the algorithm-allotted width. Mitigation: phase 1 includes a small spike that compares Q1's atom-width to `byWordWrapping`'s actual minimum-line width on a stress fixture; if they diverge, document and refine. Cheap; ~30 min.
2. **Performance regression.** Adding a per-cell min-content measurement doubles measurement cost. Mitigation: same content-hash cache covers both; first-render budget is the same; resize budget unchanged (cache hits).
3. **Q3 floor edge case.** If a column's `min_width` < 60pt and the floor pushes the column over `min`, slack distribution math could under-credit other columns. Mitigation: post-pass clamp lives outside the slack math; slack distribution always uses raw `min`. The floor only affects very narrow content (single-glyph columns), which dogfood doesn't exercise.
4. **Test fixture rewrites.** All 15 D24 unit tests change in expected values. Risk: I introduce a regression while transcribing. Mitigation: derive expected values from the published CSS algorithm and double-check with a hand-trace per test case in the COMPLETE doc.
5. **D24 manual test plan numbers shift.** Documented widths in `d24_responsive_table_columns_manual_test_plan.md` (e.g., Decision = 705.4pt at viewport=920) will change because the new algorithm distributes differently. Plan: update those numbers in the manual test plan as part of phase 4 close-out.
