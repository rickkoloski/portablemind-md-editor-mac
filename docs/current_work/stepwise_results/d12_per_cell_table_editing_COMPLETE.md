## What shipped

GFM tables now support **per-cell editing** as the primary interaction:

- **Single-click** in a cell places the caret inside that cell at a source offset within the cell's content range. Grid stays rendered. No mode switch.
- **Typing** at the caret writes to the cell's source range; the cell re-renders from storage with the edit reflected.
- **Tab** / **Shift+Tab** cycle through cells across rows (next/prev cell of same row, then first/last cell of next/prev row).
- **Left** / **Right Arrow** at cell-content boundaries jump directly to the adjacent cell, skipping the pipe + whitespace source between them.
- **Backspace** at a cell's content-start moves the caret to the previous cell's content-end without deleting anything (Word/Docs cell-boundary semantics). **Delete** is symmetric.
- **Selection highlighting** renders per-cell — multi-cell or multi-row selections produce one highlight rect per cell intersection; inter-cell pipe + whitespace is not highlighted.
- **Double-click** on a cell drops the row's table to whole-row source-reveal mode (D8.1's pipe-source view, retained as the explicit power-user trigger). Single-click no longer reveals.
- **Caret + selection-highlight x positions are CT-glyph-advance-aware** — accurate for proportional fonts and Unicode characters with non-uniform advance widths.

## Architectural findings (production-relevant insights from spike)

The spike (`spikes/d12_cell_caret/`) validated each of these before the production work began. Production preserves them.

1. **`NSTextLineFragment` cannot have arbitrary bounds.** Original §3.4 hypothesis (override `textLineFragments` with custom typographic bounds) was dead on header reading — `typographicBounds` and `glyphOrigin` are readonly. Replaced with custom `NSTextSelectionDataSource`, which IS the supported mechanism for caret-routing customization.

2. **`lineFragmentRangeForPoint`'s `location` argument is the container anchor, not the click position.** Use `tlm.textLayoutFragment(for: point)` to find the fragment hit by the click, then read `rangeInElement.location` — this is the canonical multi-row click routing approach.

3. **`enumerateCaretOffsetsInLineFragment` must yield offsets in strict left-to-right visual order.** Non-monotonic enumeration confuses NSTextView's hit-test.

4. **NSRange→NSTextRange off-by-one.** A cell's content range `(loc, len)` represents N characters, but caret positions are N+1 (before-first ... after-last). The `NSTextRange` returned from `lineFragmentRangeForPoint` must extend by +1 on length so the caret can land at the content-end position.

5. **Empty cells need both zero-length ranges + caret snap.** `parseCellRanges` records empty cells; `LiveRenderTextView.snapCaretToCellContent` corrects clicks that the click-routing pushed past the empty cell's content into pipe/whitespace.

6. **Cell ranges from `parseCellRanges` are document-absolute already.** No row-local offset shifting needed in production (different from the spike, where the fragment received row-local source).

7. **`storage.edited(.editedAttributes, ...)` inside `beginEditing/endEditing` is required to evict cached fragments** — `invalidateLayout(for:)` alone keeps the cache. Same finding as D8.1.

8. **CT glyph advances vary** for proportional fonts. Uniform-stride approximation produces visible drift for caret/highlight positioning in non-monospaced cell content. `CTLineGetOffsetForStringIndex` is the correct primitive.

## Test harness — meta-infrastructure

Production gained a `#if DEBUG`-only test harness (`Sources/Debug/`) that mirrors the spike's: an external driver writes JSON action files to `/tmp/mdeditor-command.json`, the app polls them every 200ms, executes the action, and writes results to known paths. Combined with `cliclick` (synthetic mouse) and `osascript` (synthetic keys), Claude Code can drive the running app autonomously without depending on the app being frontmost or on simulated input alone.

This harness was the inflection point in iteration velocity — see commit `2af00c5`. All D12 step validations from step 1 onward used it. Reusable for future deliverables; strip whenever no longer wanted via `grep -rn 'TEST-HARNESS:' Sources/`.

## Deferrals / known gaps

| Gap | Disposition |
|---|---|
| Inline markdown formatting inside cells (bold/italic/code/link) | Future deliverable. Cell content remains plain text from GFM source substring (D8 deferral, unchanged). |
| **`CellRenderer` protocol** for pluggable cell renderers (per spec §4) | Future. Current code uses a single default renderer. |
| Multi-line wrapped cell content | Works (CT layout handles it within the cell's `contentWidth`); caret navigation Up/Down within a wrapped cell relies on NSTextView's natural line-fragment behavior. |
| Drag-select via mouse (vs. setting selection programmatically) | Should work — selection-changed flow is unchanged from D8. Manual test plan §E covers the harness-driven case. |
| Pipe-typing UX policy | Currently allows pipe input → row gains a new column structurally. Spec listed three options (allow / auto-escape / block); allowed is current default per markdown semantics. Can be revised in a follow-up. |
| Default NSTextView selection-highlight bleed across pipe characters | Cosmetic. Production renders per-cell highlights correctly in cells; any default highlight in inter-row source is acceptable. Suppression via override is a future polish. |
