## Summary

Replaces D24's "lock-in by equal-share + flex-by-natural" distribution with the canonical CSS Tables Â§3.9 slack-proportional algorithm, plus a deliberate divergence (Q8) that pre-locks narrow data columns at `max` regardless of slack â€” so dates, IDs, statuses, and short owners stay at their content width even at narrow viewports while long-text columns absorb the squeeze. Closes the regression Rick reported on D24's ship day where Table A's Date column wrapped "2026-04-28" mid-string when the editor was narrow.

A latent rendering bug surfaced during phase 3 smoke and was fixed in the same deliverable: the `NSTextContainer.lineFragmentPadding` (default 5pt Ă— 2 = 10pt) was eating into each cell's text area at render time, but D24's natural-width measurement didn't account for it. D24's lock-in always gave short columns â‰Ą3pt of headroom which masked the discrepancy; D24.2's Q8 locks AT max with no headroom, so the gap bit. Bumped `cellFramingOverhead` 14 â†’ 24 and pad cell `contentWidth` by 2 Ă— `lineFragmentPadding` so the algorithm-applied width equals actual usable text area.

- **Q1 token-split heuristic** (`cellMinContentWidth`) â€” split on whitespace + ASCII soft-break punctuation (`-`, `/`, `.`); take widest atom's CT-shaped width per cell, max across the column.

- **Three-regime distribution** â€” `fits` (ÎŁmax â‰¤ viewport), `slack` (between), `overflow` (ÎŁmin â‰Ą viewport).
- **Q8 narrow-column threshold lock-in** â€” pre-lock any column with `max â‰¤ narrowThreshold` (default 120pt) at max, with the "leaves room for remaining mins" constraint to prevent over-locking. Greedy iterate (smallest-first) until fixpoint.
- **Slack-proportional flex** â€” flex columns get `min` + share of `surplus = pool - ÎŁ flex_min` proportional to `slack = max - min`. Degenerate fallback (totalSlack=0): split surplus equally.

## Deviations from Spec

### 1. lineFragmentPadding compensation added in phase 3 (i06)

The spec frame anticipated only the Q8 + slack-proportional algorithm change. Phase 3's smoke test surfaced a latent rendering bug: cell `contentWidth` doesn't equal the cell's usable text area because `NSTextContainer.lineFragmentPadding` (default 5pt Ă— 2 = 10pt) eats into the content area at render time. D24's lock-in algorithm always gave columns â‰Ą3pt of headroom which kept this hidden; D24.2's Q8 locks columns AT max with no headroom, so columns near the wrap threshold flickered between fitting and not-fitting during drag-resize. Fixed in `TK1TableBuilder` by:
- Splitting `cellFramingOverhead` into `cellBorderPaddingOverhead` (14pt: border + padding Ă— 2) + `2 Ă— cellLineFragmentPadding` (10pt). Total 24pt.
- Padding the cell's `contentWidth` by `2 Ă— cellLineFragmentPadding` in `makeCell` so the usable text area equals the algorithm-applied column width.

Filed as i06; not a separate deliverable since the fix is small (~5 lines) and inseparable from D24.2's regression evidence.

### 2. Spike file kept (not deleted)

Plan Â§Phase 4 said the token-split spike file would be deleted in phase 4. Kept instead at `UnitTests/D24_2/TokenSplitSpike.swift` as a regression guard for `cellMinContentWidth`. Future changes to Q1's break-character set or atom-extraction logic will surface here. Cost: ~100 lines + 0.001s test runtime.

### 3. NSLayoutManager-at-width=1pt comparator dropped

Plan Â§Phase 1 originally proposed comparing Q1's atom widths against TextKit's `byWordWrapping` minimum-line-fragment width via NSLayoutManager laid out at 1pt width. The first attempt crashed during `ensureLayout` (probable layout-loop edge case at the constrained width). Replaced with hand-derived expected atom widths per fixture. The simpler validator is sufficient for the spike's purpose. If a future regression argues TextKit's break opportunities diverge from Q1, re-attempt the layout-based comparator at moderate width like 50pt rather than 1pt.

## Notes

- **Q8 vs literal CSS spec.** D24.2 deliberately diverges from strict CSS Tables Â§3.9 for the data-editor case. Browsers/VS Code wrap dates at hyphens at narrow viewports because their min-content algorithm follows TextKit's break opportunities exactly; we choose to keep narrow data columns at natural width even at the cost of bloating the long-text column's wrap. Tunable so the divergence can be dialed back.
- **D24's Q8 vs D24.2's Q8.** D24's spec used "Q8" for the viewport-cap on natural width (preserved unchanged). D24.2's Q8 is the narrow-column threshold lock-in. Both Q8s coexist; commit messages and the spec call out the disambiguation.
- **The spike's value beyond validation.** Token-split spike's 0.0% delta on every fixture means Q1's punctuation set ({whitespace, `-`, `/`, `.`}) is a near-perfect approximation of what dogfooded markdown content actually atomizes to. Broadening to `,`, `;`, `:` etc. is unnecessary â€” those characters rarely appear inside atomic content units.
